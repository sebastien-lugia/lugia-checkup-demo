# Instruction générale du dossier — Lugia Check-up Demo

Tu travailles dans le dossier local du projet `lugia-checkup-demo`.

Ce projet vise à construire progressivement un démonstrateur local Streamlit pour Lugia : un check-up préventif B2B permettant à un médecin généraliste de répondre à la question : “où en est réellement mon cabinet aujourd’hui ?”

Ne cherche pas à tout construire d’un coup. Avance phase par phase, en produisant uniquement les fichiers ou modifications nécessaires à l’étape en cours. Avant chaque phase, explique brièvement ce que tu vas faire. Après chaque phase, indique comment tester localement, ce qui a été modifié, et ce qui reste à valider.

Le projet doit rester simple, lisible et itérable. Privilégie une V1 fonctionnelle plutôt qu’une architecture trop complexe. Si une fonctionnalité devient trop lourde pour la V1, propose une version simple et ajoute l’amélioration à `ROADMAP.md`.

Les ressources Markdown du dossier sont la mémoire produit et méthodologique du projet. Consulte-les avant de modifier le code. Maintiens une cohérence entre le code, les ressources, les templates de sortie, le scoring, les chantiers et le positionnement Lugia.

Respecte le positionnement Lugia : ton simple, clair, humain, professionnel, non culpabilisant. Le produit analyse le système de travail, pas les individus. Il ne doit jamais produire de diagnostic médical, ni demander ou stocker de données patient identifiables.

À chaque modification structurante, mets à jour les fichiers de suivi pertinents :
- `CHANGELOG.md` pour les changements réalisés ;
- `TODO.md` pour les tâches restantes ;
- `DECISIONS.md` pour les décisions importantes ;
- `ROADMAP.md` pour les améliorations repoussées.

Lorsque tu modifies un fichier, évite de réécrire inutilement tout le projet. Travaille proprement, en gardant une arborescence claire, des noms explicites, du code compréhensible et des commentaires utiles seulement quand ils apportent de la clarté.
